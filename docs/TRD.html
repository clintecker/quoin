TRD Quoin · v0.1 draft 2026-07-02

Quoin — Technical Design

Fully native, zero JavaScript, one shared core — built to grow an editor.

Scope update (2026-07-02): per the design handoff in docs/design/handoff.md, Quoin is now a WYSIWYG editor. The architecture here stands — the editor is a new mutation source feeding the DocumentSession pipeline (§06), exactly as designed. The handoff's constraints (AST as sole source of truth, NSDocument autosave, byte-lossless round-trip) are binding.

01Architecture at a Glance

Four decisions define the system; everything else follows from them:

  • Native rendering, not a web view. The document is displayed with TextKit 2 attributed text, not HTML in a WKWebView. This is what makes launch, scroll, and re-render genuinely fast and the app feel native (selection, accessibility, dark mode, Dynamic Type for free).
  • Zero JavaScript, anywhere. Math and Mermaid are rendered by in-house native engines (QuoinMath, QuoinDiagram, §04) drawing with CoreText/CoreGraphics — no WKWebView, no bundled JS, no runtime it can't account for. Diagram support ships in phases; anything not yet supported degrades to the styled source block, never breaks.
  • One dependency for parsing. Apple's swift-markdown (which wraps the reference C parser cmark-gfm) provides spec-complete CommonMark + GFM parsing with source ranges. Writing our own spec-complete parser is months of work for a worse result; this is the one dependency that earns its place — Apple-maintained, tiny, no transitive deps.
  • Read-optimized now, edit-ready later. V1 is a viewer, but the document pipeline is a session that accepts source mutations (today: file changes and checkbox toggles; tomorrow: keystrokes from an edit pane) and emits immutable rendered snapshots. An editor becomes a new input source, not a rewrite (§06).

02Document Model (QuoinCore)

The core type is QuoinDocument: an immutable value holding the source text, the parsed block tree, and a source map (every node knows its exact UTF-8 byte range in the source). The source map is load-bearing — it powers checkbox write-back, search-result → scroll mapping, incremental re-render diffing, and stable scroll anchoring across reloads.

struct QuoinDocument {
    let source: String
    let blocks: [Block]            // stable identity via content hash + position
    let outline: [Heading]         // TOC, precomputed
    let stats: DocumentStats       // words, chars, reading time, tasks…
}

enum Block: Identifiable {
    case heading(level: Int, inline: [Inline], range: SourceRange, slug: String)
    case paragraph([Inline], SourceRange)
    case codeBlock(language: String?, code: String, SourceRange)
    case mermaid(source: String, SourceRange)          // recognized fence
    case mathBlock(latex: String, SourceRange)         // $$…$$
    case table(header: [TableCell], rows: [[TableCell]], alignments: [Alignment], SourceRange)
    case list(items: [ListItem], ordered: Bool, SourceRange)   // items carry .task(checked:)
    case blockQuote([Block], SourceRange)
    case thematicBreak(SourceRange)
    case htmlBlock(String, SourceRange)                // shown as literal, v1
}

Parse pipeline: cmark-gfm parse → walk into Block tree → two lightweight post-passes: (1) fence dialect detection (```mermaid → .mermaid), (2) math span extraction ($…$/$$…$$ with standard escaping rules) since cmark has no math extension. All pure Swift, all off the main thread.

Concurrency: parsing, stats, and export run on a background actor; the UI receives complete immutable QuoinDocument values. No locks, no partial states on screen.

03Rendering (QuoinRender)

Each Block renders to an NSAttributedString via a theme (fonts, spacing, colors resolved from the system appearance). The document view is a TextKit 2 NSTextLayoutManager stack with viewport-based layout — TextKit 2 lays out only what's visible, which is what keeps a 500-page document scrolling at full frame rate.

  • Tables get a custom layout pass (attributed-string tables are too limited): measured column widths, header row emphasis, horizontal scroll for wide tables — rendered as an embedded native view attachment.
  • Code blocks use a small native regex/state-machine highlighter (~15 common languages) — deliberately not a full grammar engine; good-enough highlighting at zero dependency cost.
  • Images resolve relative to the document, decode async at display size (CGImageSource downsampling), and fill a placeholder attachment sized from image metadata to avoid layout jumps.
  • Math & Mermaid render as placeholder attachments immediately; QuoinMath/QuoinDiagram fill them in asynchronously (see §04). Cached content appears on first layout with no flash.
  • Checkboxes are interactive attachments; toggling calls the write-back service (§06).

Live update: on file change, re-parse (cmark parses 1 MB in single-digit ms — full re-parse is simpler and fast enough), then diff old/new block lists by identity. Only changed blocks re-render; unchanged attachment islands (diagrams, images) are reused from cache. Scroll position is re-anchored to the nearest stable block, so a save near the bottom doesn't yank the view.

04Native Math & Diagram Engines

Both engines are in-house Swift packages that draw with CoreText/CoreGraphics into resolution-independent layers, sit behind a common BlockArtist protocol ((source, theme, width) → CALayer/Path tree), and share a content-hash render cache. They render asynchronously into placeholder attachments; failures and unsupported input always degrade to the styled source block with a caption — a document never breaks.

QuoinMath — TeX-subset typesetter

  • Classic TeX boxes-and-glue layout: parse LaTeX math into a node tree (symbols, fractions, scripts, radicals, delimiters, accents, matrices/aligned environments, operators with correct spacing classes), typeset using the math table of a bundled OpenType math font (STIX Two Math, OFL — a font asset, not a code dependency).
  • Coverage target: the LaTeX-math subset that covers ~99% of real markdown documents (KaTeX's common set). Unknown commands render as literal code with a subtle marker rather than erroring.
  • Inline math participates in line layout with correct baselines; display math centers as a block.

QuoinDiagram — native Mermaid engine

A Mermaid-dialect parser plus per-diagram layout engines, shipped in phases; each phase is spec'd against Mermaid's own test corpus for visual fidelity where it matters (topology, labels, direction) rather than pixel parity.

05File Watching & Live Rendering

  • DispatchSource.makeFileSystemObjectSource on the open file descriptor (catches writes, renames, atomic-save replace patterns) with an NSFilePresenter fallback for sandbox-mediated coordination on iOS.
  • Change events are debounced (~50 ms) and coalesced; reload reads, re-parses, block-diffs, and patches the layout — target <100 ms save-to-screen.
  • Follow mode (P1) pins the viewport to the end when the previous scroll position was at the end — the "watch AI output stream in" case.

06Document Session & Write-Back (edit-ready)

Each open document is owned by a DocumentSession actor — the single authority over the source text. It accepts source mutations from any input and runs each through the same pipeline: apply to source → re-parse → block-diff → publish a new immutable QuoinDocument snapshot.

inputs                       DocumentSession                outputs
─────────                    ───────────────                ────────
file watcher ─┐              source of truth: text          new QuoinDocument
checkbox tap ─┼─ mutation →  apply · reparse · diff  ──→    snapshots → UI,
edit pane*   ─┘              coordinate file I/O            search, stats, TOC
                                                            (* future)
  • V1's only user-initiated mutation is the checkbox toggle: using the task item's SourceRange, replace exactly the [ ]/[x] marker bytes — nothing else in the file is touched, formatting preserved byte-for-byte. Writes go through NSFileCoordinator; the resulting file event is recognized as self-inflicted (content hash) and skipped.
  • Conflict rule: if the file changed on disk since last parse, re-parse first and re-locate the target node by identity before writing.
  • Future edit mode drops in here: an editor pane is just another mutation source feeding ranged text edits into the session — the render path, diffing, exports, and file coordination are already built for it. Undo stacks on the mutation log. Nothing in QuoinRender assumes documents are static; it already re-renders diffs continuously for live updates.

07Search, TOC, Statistics

  • ⌘F search: case/diacritic-insensitive scan over the plain-text projection of the document (precomputed with per-block offsets). Matches map through the source map to layout ranges for highlight overlays and jump-to. Incremental as-you-type; a 1 MB doc scans in well under a frame.
  • TOC: the precomputed outline renders in a collapsible sidebar; scroll-spy via TextKit viewport callbacks; heading slugs (GitHub algorithm) also serve #anchor links.
  • Stats: computed during the parse pass (words via enumerateSubstrings(.byWords) — Unicode-correct, CJK-aware; reading time at 230 wpm; tasks done/total; headings, links, images, code blocks). Live-updates with the document.

08Export Pipeline

All exporters are pure functions from QuoinDocument in QuoinCore — the same output on macOS and iOS.

09App Shell

  • macOS: document-based app (NSDocument-backed, read-only document class) — this buys native window tabs, Open Recent, restoration, and UTI registration (net.daringfireball.markdown) for free. SwiftUI UI hosted in AppKit windows; the document view itself is an NSTextView (TextKit 2 mode) wrapped for SwiftUI.
  • iOS/iPadOS: DocumentGroup + the system document browser; same QuoinRender view via UITextView; TOC in a sidebar (iPad) / sheet (iPhone); exports through the share sheet. iPad gets multi-window scenes in place of tabs.
  • Minimalist UI: one toolbar (TOC toggle · title · search · stats · export). Panels closed by default; everything keyboard-reachable.
  • Sandbox: full App Sandbox, user-selected-file read access (write only for the checkbox path + security-scoped bookmarks for relative images/links in the document's folder). No network entitlement usage at runtime.

11Testing & Performance Verification

  • Spec conformance: run the CommonMark spec suite + GFM extension tests against the parse→AST layer; golden-file tests for the math/mermaid post-passes.
  • Render snapshots: attributed-string snapshot tests per block type and theme (light/dark); image-snapshot suites for QuoinMath (against a LaTeX corpus) and QuoinDiagram (against Mermaid's example corpus).
  • Exports: golden PDFs/RTF for a corpus of representative documents.
  • Write-back: property tests — toggling any checkbox changes exactly the marker bytes; round-trip byte equality elsewhere.
  • Performance: XCTest measure baselines wired to the PRD budgets (cold open, 1 MB parse, save-to-screen latency, scroll hitch rate via XCTOSSignpostMetric); CI fails on regression.
  • Corpus: a checked-in torture-test folder — huge tables, 10k-line files, pathological nesting, emoji/CJK, broken mermaid, missing images.
